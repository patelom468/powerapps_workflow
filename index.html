Overview

The Problem & The Solution

A manual, paper-based internal approval process was causing bottlenecks. Forms were lost, approvals were delayed, and there was no visibility into where a request stood. This project replaced it entirely with a structured digital workflow.

🔴 The Problem
  • Physical paper forms were regularly misplaced or delayed in transit between departments
  • No central record of approval status — staff had to chase approvers manually
  • Errors from illegible handwriting led to rejected submissions being re-processed
  • No automated notifications — approvers often weren't aware of pending requests
  • Reporting on approval rates was nearly impossible without manual tallying
🟢 The Solution
  • Canvas app built in Power Apps with validated form fields and dropdown menus
  • Requests stored in SharePoint — always accessible, searchable, and backed up
  • Power Automate flow sends instant email notifications to approvers on submission
  • Approvers review and action requests directly in the app — one click to approve or reject
  • All status changes logged with timestamps — full audit trail for compliance
Process Design

How the Workflow Works

Five automated steps — from submission to archive — without a single manual hand-off.

01
Request Submission
A staff member opens the Power Apps canvas app and fills out a structured request form. Required fields are validated before submission — no incomplete forms reach the queue. The request is saved to a SharePoint list with a status of Pending.
Power Apps Canvas App
02
Automated Notification
A Power Automate cloud flow is triggered instantly when a new item is added to the SharePoint list. It sends a formatted email to the designated approver, including a direct link to the request record in SharePoint and a summary of the submission details.
Power Automate · Email Notification
03
Approver Review
The approver opens the app (or SharePoint directly) to review the request in full. A role-based view ensures only authorised approvers see the approval action buttons. They can approve, reject, or flag for follow-up — each action logs the approver's name and a timestamp automatically.
Role-Based Access · SharePoint
04
Decision Notification
A second Power Automate flow triggers on status change and emails the original requester with the outcome — approved, rejected, or pending additional information. Rejected requests include the approver's optional notes field.
Power Automate · Conditional Logic
05
Archive & Reporting
Completed requests are automatically moved to an archive SharePoint list. A Power Apps dashboard shows real-time counts of pending, approved, and rejected requests — eliminating the need for manual reporting spreadsheets entirely.
SharePoint · Power Apps Dashboard

Technology

Tools Used

Microsoft's Power Platform ecosystem — chosen for rapid development, zero infrastructure cost, and seamless integration with the organisation's existing Microsoft 365 environment.

⚡
Power Apps
Canvas app for the submission form and approval queue. Responsive layout that works on desktop and mobile.
🔄
Power Automate
Two cloud flows: submission trigger → approver email, and status change trigger → requester email. Conditional branching for approve/reject paths.
📁
SharePoint
Data layer — two lists: active requests and archived requests. Column-level permissions for role-based access.
📧
Outlook / Email
HTML-formatted email notifications sent via Power Automate's built-in Office 365 Outlook connector.
👤
Azure AD / M365
Authentication and user identity handled by Microsoft 365 — no separate login system needed.
📊
Power Apps Dashboard
Live chart and count display for pending, approved, and rejected requests — no manual reporting.

Reflection

What I Learned

1
Low-code doesn't mean low-thinking. Designing the data model in SharePoint and the conditional logic in Power Automate required the same careful thinking as writing custom code. The tool is different; the problem-solving process isn't.
2
Understanding the existing process is the real work. The hardest part wasn't building the app — it was mapping every edge case in the paper process and deciding how to handle them digitally. Discovery interviews with actual users were invaluable.
3
Validation prevents downstream problems. Adding required field validation and dropdowns instead of free-text entry eliminated an entire category of errors — incomplete or ambiguous submissions that previously required rework.
4
Power Automate's error handling matters. Testing flows with deliberate failures (e.g. email address not found, SharePoint permission denied) revealed gaps in the error path. Robust flows always account for what happens when things go wrong.
5
The right tool for the right job. A custom-coded solution would have taken weeks and required ongoing maintenance. Power Platform delivered a working, maintainable solution in days — and fits perfectly within a Microsoft 365 organisation.
